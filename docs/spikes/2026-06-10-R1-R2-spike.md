# Spike: R1 Vercel runtime metrics + R2 Neon live connections (2026-06-10)

Discovery only. Five parallel doc-research agents over live vercel.com/docs, neon.com/docs, both OpenAPI specs (openapi.vercel.sh fetched + grepped; neon.com/api_spec/release/v2.json downloaded + grepped), vercel/vercel CLI source. One real authed probe via Vercel MCP.

## Verdicts

**R1 — cold starts programmatically: NO.** "Start type" (Hot / Hot prewarmed / Cold) exists in dashboard Runtime Logs UI + Observability "Cold start frequency" tile only. Absent from: documented REST API (full spec grepped — 0 hits for cold/coldStart/startType), runtime-logs endpoint fields, Drains log schema v1, Trace Drain attrs, Observability query-builder fields. Exposing it in drains = open feature request since Jan 2025, no commitment.

**R1 — invocations / error rates / percentiles: YES**, two channels:
1. **Observability Plus query API** — `vercel metrics` CLI (public beta 2026-05-05) backed by **undocumented** `POST /v2/observability/query` + `GET /v2/observability/schema` (verified in CLI source). Metrics: request count, function invocations, durations, CPU/wall time; aggregations incl. p75/90/95/99; group-by route/project/status; filter OData. Granularity 1m (≤1h window) → 5m/15m/1h beyond. Freshness undocumented. Requires **Observability Plus**: paid Pro/Enterprise add-on, $1.20/1M events, 30d retention. Error rate = request count grouped by HTTP status (no dedicated error metric).
2. **Vercel Drains** (supersedes Log Drains, 2025-09): POST JSON/NDJSON to any HTTPS endpoint, batched, ~seconds latency (unofficial), 100% unsampled by default, HMAC-**SHA1** `x-vercel-signature`. Log schema has requestId, path, statusCode (-1 = crash), executionRegion, proxy.* (cache, pathType, lambdaRegion, WAF), traceId/spanId. **No coldstart, no duration, no memory.** `type:"report"` entries exist but contents undocumented (Fluid Compute likely killed classic REPORT semantics). Plan: **Pro/Enterprise only**, $0.50/GB. Trace Drains: OTLP/HTTP (json or protobuf) to arbitrary endpoint, infra+fetch spans, attrs only vercel.projectId/deploymentId.

**R2 — connection counts without connection string: YES, Scale plan only.** Neon metrics export pushes **`neon_connection_counts`** (gauge, labels state=active|idle, project_id/endpoint_id/compute_id) + **`neon_pgbouncer_pools_client_active/waiting_connections`**, `..._server_active/idle_connections`, maxwait, host CPU/mem, LFC, replication lag — via per-project OTel collector, 30s internal scrape, OTLP over **HTTP or gRPC, push, to any base URL we host** (collector appends /v1/metrics, /v1/logs), auth = bearer/basic/custom header. Gotchas: **Scale plan only** (not Free/Launch); **console-only setup** — no public API to enable (TelemetryConnection/TelemetryConfig schemas exist in OpenAPI spec but referenced by zero paths ⇒ programmatic config likely imminent, not today); suspended compute sends nothing (gap ⇒ 0 conns, acceptable). We CAN detect integration presence read-only via `GET /projects` → integrations map (vercel|github|datadog|opentelemetry).

**R2 — pg_stat_activity: YES, any plan, needs connection string + cost-aware polling.**
- Privileges: Neon console-created roles are `neon_superuser` members ⇒ hold `pg_monitor` WITH ADMIN OPTION ⇒ customer can: `CREATE ROLE monitor LOGIN PASSWORD '...'; GRANT pg_monitor TO monitor;`. Total counts + per-user/db counts work even unprivileged; state breakdown needs pg_monitor.
- Sees Postgres-side connections only. PgBouncer client fan-in (up to 10k) invisible to SQL — no admin console, no SHOW POOLS. Pooler stats = console graphs or OTel export only.
- **Cost trap: every connection resets the 5-min autosuspend timer. 60s polling = compute awake 24/7** ≈ 182.5 CU-hr/mo at 0.25 CU ≈ $19/mo Launch, $40/mo Scale, exhausts Free's 100 CU-hr in ~17 days. Neon documents the same effect for pgAdmin.
- Mitigation (design): **event-gated polling** — poll only when Vercel-side signals live traffic (drain entries / deploy events). Compute is already awake when traffic flows ⇒ zero added cost; idle ⇒ don't poll ⇒ no data needed (0 conns).
- Neon REST API itself: NO connection metrics anywhere. Endpoint state (init|active|idle) + last_active + consumption history (hourly granularity, ~15-min lag, Launch+ plans, doesn't wake compute).

## Headline demo (coldstart → pool exhaustion): NO as stated. YES reframed.

Literal version dead: no programmatic cold-start signal exists on any plan.
- **Reframe A (recommended): deploy → Neon connection spike.** All signals available: deploy webhook (free) + `neon_connection_counts`/pgbouncer waiting (Scale OTel) or gated pg_stat_activity. Same wow, same correlation shape.
- **Reframe B: invocation surge → pool exhaustion.** Obs Plus request-count @1m granularity vs Neon connections. Cold-start storm is *implied* by surge-after-deploy + p99 spike (heuristic, label as such).

## Architecture deltas (apply at schema freeze)

1. **New ingest surface**: `/api/ingest/drain` (Vercel Drains NDJSON; HMAC-SHA1 verify — NOT SHA256 as §5 of ARCHITECTURE.md implied) and `/api/ingest/otlp` (OTLP/HTTP receiver for Neon push; accept protobuf + JSON). MetricPoints from push, not just poll.
2. **poll-vercel** = Observability Plus query client against undocumented `/v2/observability/query`: pin request shape, feature-flag, degrade gracefully (it can break without notice).
3. **Neon tiers**: Scale ⇒ OTel push (guided console walkthrough, we issue per-connection ingest token); non-Scale ⇒ consent-gated `pg_monitor` role + event-gated SQL polling; floor ⇒ consumption API (hourly, 15-min lag) + endpoint state poll.
4. **Taxonomy**: drop `vercel.function.coldstarts` from v1 metrics. Add `neon.pgbouncer.client_waiting`, `neon.pgbouncer.server_active`, `vercel.request.count`, `vercel.request.p99ms` (Obs Plus), TTFB via `analytics v2`/`speed_insights v1` drain schemas (bonus effect signals).
5. **Customer plan floor** (onboarding must state): Vercel Pro (+ Drains $0.50/GB, Obs Plus $1.20/1M events) and Neon Scale (or conn-string consent). Free/Hobby users get webhooks-only degraded mode.
6. Seed rule update: replace coldstart rules with `vercel.deployment.succeeded → metric:neon.connections.active:spike` and `metric:vercel.request.count:spike → metric:neon.pgbouncer.client_waiting:spike`.

Empirical note: authed probe on user's own Vercel team (Hobby) — runtime logs API reachable, no drains/Obs Plus available on that tier, consistent with gating above.
